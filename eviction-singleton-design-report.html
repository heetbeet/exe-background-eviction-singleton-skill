ExeWrap Eviction Singleton Design

This is a deliberately lightweight, best-effort pattern. It is not a strict singleton and does not use a mutex, PID file, secondary lock, or a general service manager.

Can ExeWrap identify itself by full path?

Yes, for this purpose. ExeWrap exposes its exact current path as @{exe_path}. Stamp that into an environment variable such as SINGLETON_OWNER_EXE, then have the launched PowerShell supervisor query Win32_Process.ExecutablePath for running PIDs and compare canonical full paths case-insensitively.

Use ExecutablePath, not the command line. The command line may contain a bare filename, altered quoting, or no launcher path at all. Windows records the executable image path separately. This distinguishes two start-background.exe files in different directories.

Chosen model: eviction singleton

  1. A stamped, windowed ExeWrap executable starts a supervisor.
  2. The stamp injects its own full path using @{exe_path}.
  3. The supervisor finds prior processes whose ExecutablePath equals that exact path.
  4. It forcefully terminates each matching process tree.
  5. It starts the new worker, with ExeWrap's Job Object retaining child-tree cleanup.

This accepts a startup race: two nearly simultaneous starts may evict each other briefly. That is acceptable under the stated goal—new launches evict old same-path launches—without adding persistent coordination.

What this deliberately excludes

  • No instance_id or stamped unique identifier.
  • No mutex, PID/state file, lockfile, or registry state.
  • No graceful shutdown protocol; forceful tree termination is the chosen policy.
  • No matching by executable filename, port, command-line fragments, or app-specific worker names.
  • No Startup registration, scheduled tasks, dependency installation, or application supervision policy.
  • No automatic “latest release” fetching. The user of the skill chooses a current pinned ExeWrap release and verifies its published checksum.

Boundaries

The ownership boundary is “same canonical stamped-executable path,” not a stronger security identity. It safely avoids collisions with identically named executables elsewhere, but any process actually launched from that same executable path is intentionally eligible for eviction. Reading ExecutablePath can also be restricted for processes owned by another user or elevated context; this pattern is designed for the normal same-user background-launcher case.

Result

The accompanying skill is named exe-background-eviction-singleton. It provides an ExeWrap config template and a PowerShell eviction helper that implements this exact scope.
